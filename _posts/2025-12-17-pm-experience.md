## 프로젝트 개요

8개월간 견적 시스템 신규 구축 프로젝트에서 PM(Project Manager) 역할을 수행했습니다. PM과 백엔드 개발을 병행했습니다.

**주요 업무:**
- 고객 요구사항 정리 및 분석
- 개발팀 업무 분배 및 일정 관리
- 고객사 커뮤니케이션
- 이슈 대응 및 리스크 관리

## PM의 핵심 업무

### 1. 요구사항 관리

#### 문제: 변경되는 요구사항

고객의 요구사항이 프로젝트 중간에 자주 변경되는 문제가 있었습니다.

#### 해결: 회의록 기반 관리

모든 회의 내용을 문서화하고 고객 확인을 받았습니다.

**회의록 템플릿:**
```
# 회의록

**일시:** 2024.05.10 14:00~15:30
**참석자:** 고객사 이사님, 팀장님, PM

## 논의 사항
1. A 기능 추가 요청
   - 내용: ...
   - 예상 공수: 3MD
   - 일정 영향: +1주

2. B 기능 변경 요청
   - 기존: ...
   - 변경: ...
   - 예상 공수: 2MD

## 결정 사항
1. A 기능은 Phase 2로 연기
2. B 기능은 X 방식으로 개발
3. C 기능은 다음 주까지 완료

## Action Items
- [ ] PM: 변경된 일정표 공유 (~ 05.12)
- [ ] 개발팀: B 기능 상세 설계 (~ 05.15)
- [ ] 고객사: C 기능 테스트 데이터 제공 (~ 05.13)
```

**효과:**
- "제가 그런 적 없는데요?" 방어 가능
- 결정 사항 추적 가능
- 이슈 발생 시 근거 자료로 활용

### 2. 일정 관리

#### 버퍼 타임 설정

개발자의 예상 공수에 여유를 추가했습니다.

**공식:**
```
실제 일정 = 개발자 예상 공수 × 1.5 ~ 2.0
```

**예시:**
- 개발자 예상: 3일
- PM 일정 수립: 5일
- 이유: 예상치 못한 이슈, 테스트 시간, 리뷰 시간 포함

#### Gantt Chart 작성

Confluence로 작성했습니다.

```
Task                | 담당자 | 시작일   | 종료일   | 진행률
-------------------|--------|----------|----------|-------
요구사항 분석      | PM     | 04.01    | 04.10    | 100%
DB 설계            | PM     | 04.08    | 04.15    | 100%
화면 설계          | PM     | 04.11    | 04.20    | 100%
로그인 API 개발    | 개발A  | 04.16    | 04.22    | 80%
견적 조회 API      | 개발B  | 04.16    | 04.25    | 60%
...
```

### 3. 커뮤니케이션 전략

#### 고객사 커뮤니케이션

**문제 상황:** "이 기능 추가하는 거 어렵나요?"

**나쁜 답변:**
- "어렵습니다" → "실력이 부족한 거 아니냐"
- "쉽습니다" → "그럼 빨리 해주세요"

**좋은 답변:**
```
"해당 기능은 다음 작업이 필요합니다:
1. A 모듈 수정 (2일)
2. B API 개발 (3일)
3. 기존 C 화면과 연동 (2일)
4. 통합 테스트 (1일)

총 8일(1.6주) 소요 예상됩니다.
현재 일정에 추가하시겠습니까, 다른 기능과 교체하시겠습니까?"
```

**핵심:**
- 구체적인 작업 단위로 분해
- 명확한 공수 제시
- 선택권을 고객에게

#### 개발팀 커뮤니케이션

**업무 분배 시:**
```
TO: 개발자A
SUBJECT: 로그인 API 개발 요청

[개요]
- 목적: 사용자 인증 API 구현
- 예상 공수: 3일
- 마감일: 04.22 (금)

[상세 요구사항]
1. 이메일/비밀번호 로그인
2. JWT 토큰 발급
3. 비밀번호 암호화 (BCrypt)

[제공 자료]
- ERD: [링크]
- 화면 설계서: [링크]
- API 명세서: [링크]

[확인 요청]
- 3일 공수가 적절한지?
- 추가 필요한 정보가 있는지?
```

### 4. 이슈 관리

#### 이슈 추적 템플릿

```
# ISSUE-001

**발생일:** 2024.05.15
**심각도:** High
**담당자:** 개발자A

## 문제
로그인 시 세션 타임아웃이 즉시 발생

## 영향
사용자가 로그인 상태를 유지할 수 없음

## 원인
세션 타임아웃 설정값이 1초로 되어있음 (오타)

## 해결 방안
application.yml 설정 수정
- 기존: server.servlet.session.timeout=1s
- 변경: server.servlet.session.timeout=1800s

## 일정 영향
없음 (즉시 수정 가능)

## 상태
[CLOSED] 2024.05.15 18:00
```

## PM과 개발 병행

### 시간 분배

```
09:00-10:00  일일 스탠드업 미팅
10:00-12:00  개발 (집중 시간)
12:00-13:00  점심
13:00-14:00  고객사 회의
14:00-17:00  개발
17:00-18:00  일정 관리, 이슈 정리
```

### 효율성 저하 요인

- 컨텍스트 스위칭 비용
- 회의로 인한 집중 시간 부족
- PM 업무와 개발 우선순위 충돌

### 대응 방법

1. **오전에 집중 개발 시간 확보**
2. **회의는 오후에 몰아서**
3. **긴급하지 않은 이슈는 다음 날 처리**

## 도구 활용

### 1. 이슈 추적: Jira or Redmine

```

└─ Story: 로그인 기능
   ├─ Task: 로그인 API 개발
   ├─ Task: 로그인 화면 개발
   └─ Task: 테스트
```

### 3. 일정 관리: MS Project or Google Sheets

## 주요 학습 포인트

### 1. 문서화는 필수입니다

말로만 하면 기억이 사라집니다. 모든 것을 기록해야 합니다.

### 2. 여유 있는 일정을 수립해야 합니다

딱 맞게 잡으면 반드시 밀립니다. 버퍼를 충분히 두어야 합니다.

### 3. 구체적으로 소통해야 합니다

"어렵다", "쉽다"가 아니라 작업 단위와 공수로 설명해야 합니다.

### 4. 개발 경험이 PM에 도움됩니다

개발자의 예상 공수가 현실적인지 판단할 수 있습니다.

## 결론

PM은 개발과는 다른 역량을 요구합니다:
- **커뮤니케이션**: 기술적 내용을 비개발자에게 설명
- **협상**: 일정, 범위, 품질의 균형 조정
- **문서화**: 모든 결정 사항을 추적 가능하게 기록

개발 경험이 있는 PM은 기술적 판단이 가능해 프로젝트 관리가 더 효과적입니다.
